## Summary
Overall test plan is strong, concrete, and well‑scoped. Issues mainly: minor ambiguities in thresholds, terminology consistency, explicit mapping to SSD for a few items, and overloading of certain terms ("round" vs "company_round"). Several tasks may need minor rewording to lock behavior. No blocking contradictions.

## Issues & Suggestions

### 1. Ambiguous plan list formatting
- CAT: Clarity / Format
- Location: Section 2 Objectives bullet 2
- Issue: "financial simulation plans A B C D K P R F E" lacks separators; ordering differs from later sections (A,B,C,D,K,P,R,F,E) but currently compressed.
- Suggestion: Replace with `plans A, B, C, D, K, P, R, F, E` for readability.

### 2. Snapshot float tolerance placement
- CAT: Clarity
- Location: Objectives bullet 3 & Section 5.3 last paragraph
- Issue: Two different mentions of tolerance ("avoid floating drift" vs explicit absolute diff <=1e-6 after rounding). Potential confusion whether rounding then diff or diff on raw values.
- Suggestion: Standardize: "Compare raw float values with absolute diff <= 1e-6; snapshot file stores rounded (2–4 decimals) purely for visual diff." Adjust wording in 5.3.

### 3. Settlement bonus threshold wording
- CAT: Clarity
- Location: Tasks 4 and 9 & Section 5.7
- Issue: Task 4: "after round > 15 settlement bonus deactivation" vs Task 9: explicitly lists 15 and 16; Section 5.7: "After round > 15 settlement bonus disabled". Ambiguous whether round 16 is first without bonus or if change occurs immediately after completing round 15.
- Suggestion: Clarify: "Rounds 1–15 inclusive: bonus active; rounds >=16: bonus inactive" (validate vs SSD). If SSD does not specify, tag NEED_DECISION.

### 4. Tax logic first vs subsequent rounds
- CAT: Clarity
- Location: Section 5.5
- Issue: States first round tax equals `gross * 0.033` but subsequent rounds "aggregate correctly" is vague.
- Suggestion: Specify invariant: `per_round_tax = round_gross * 0.033` and `cumulative_tax = sum(per_round_tax_i for i<=n)`; assert no tax applied to already net amounts. If compounding or alternative scheme possible, mark NEED_DECISION.

### 5. Achievement rate override behavior

- Suggestion: Define deterministic rule (e.g., longer list truncated to internal expected length N; shorter list right-padded with last provided value or zeros). If internal code not stable, mark NEED_DECISION.
